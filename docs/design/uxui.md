# UX/UIガイドライン

見た目と操作感の指針。機能仕様（画面・シーン・表示要素・操作）は [screens.md](./screens.md)、本書は「どう見せる・どう感じさせるか」（牌の描画・ハイライトの見せ方・配色・演出・レスポンシブ）。UI 未着手のため多くは仮で、実装時に詰める。「プレッシャーをかけない」（[product-concept](../product-concept.md)）を見た目・演出にも適用する。

## 1. 牌の描画

- 牌は SVG で描画（Unicode 麻雀牌文字を使わない理由は [architecture](./architecture.md) §1）。手牌・副露・上がり牌・ドラ表示牌で再利用できるコンポーネントにする。
- 上がり牌は他と区別（位置を離す／色枠 等）。副露面子は横向き等で区別。
- 赤ドラ（赤5m/5p/5s）は色で区別表示。

## 2. 場の状況の表現（リーチ 等）

「自分の情報は手牌まわり」（自風と同じ自分エリア）にまとめる。フラグはテキストのバッジに頼らず、実物の所作・道具で見せることを優先する。

- リーチ：手牌の前（自分から見て卓の中央側＝画面では手牌の上側）に千点棒を1本、横向きに置いて表す（卓上の実際の所作どおり）。点棒も牌と同様に SVG で描画（白地に赤い点。Unicode は使わない）。`WinContext.riichi` に連動。
- ダブルリーチ：同じ千点棒で表す（第一巡である旨の明示が要るなら添えるのは任意）。`WinContext.doubleRiichi` に連動。
- ツモ／ロン、一発・海底・河底・嶺上・槍槓 等その他フラグの見せ方は実装時に確定（仮）。
- ハイライト連携：採点で「リーチ」役をクリックしたとき千点棒を光らせたい。対応する `HighlightTarget`（[data-model](./data-model.md) §10）が必要になれば型側に追加する。

## 3. ハイライトの見せ方

- 採点項目クリックで対応要素を光らせる連携の動き自体は [screens.md](./screens.md) 解説シーン、光らせる対象データ（`HighlightTarget`）は [data-model](./data-model.md) §10。本書はその見た目。
- 光らせ方（枠・発光・色）はやわらかく、答えを急がせない。魔女キャラの「符（お札）を貼る」等のキャラ演出はペルソナ/描画側で乗せられる（対象データは中立）。

## 4. 配色・レイアウト

- タイポ・余白は UI 実装時に詰める。レスポンシブ（端末・向き対応）の方針は §6。
- 背景：真っ黒は使わず濃い紫紺をベース（`--color-bg`。牌の白とのコントラストは保つ）。フラットだと野暮ったいので、画面全体に淡い対角グラデ（上やや明→下暗。`--color-bg-top`→`--color-bg`）を重ねて奥行きを出す。その上に、選択中キャラのテーマ色（`Character.themeColor`。[data-model §13](./data-model.md)）で「立ち絵の背後＝画面左下のステージ帯」へ向けたラジアルグロー（＋右上に弱いアンビエント）を敷く。グラデ・グローはいずれも低アルファで、盤面（牌エリア）の白牌の可読性を守る。
- キャラ別テーマ：グロー色はキャラごと（テーマ色未指定は既定色 `defaultThemeColor` へフォールバック）。背景＝「いま誰と学んでいるか」の identity を映す（背骨：キャラ駆動）。
- 状況では色を変えない：背景グローは常に一定。誤答で赤くする等の「採点された」演出はしない（プレッシャーをかけない。[product-concept §3](../product-concept.md)）。正誤は4択パネルとキャラの表情・セリフが担う。お祝い側＝ポジティブ方向の一瞬の演出のみ将来余地（ネガティブ演出はやらない）。
- 色はデザイントークン（CSS カスタムプロパティ）に集約し直値を散らさない（[architecture §5](./architecture.md)）。

## 5. 演出・アニメ・フロー（仮）

- クイズ：回答 → キャラのリアクション → 解説、の流れを気持ちよく（[screens.md](./screens.md)）。
- 正誤・節目の演出はお祝い止まりで、プレッシャーをかけない。音まわりは [sound.md](./sound.md)。
- アニメ・演出の気持ちよさは人手で確認（[testing](../dev/testing.md) の人手領域）。

## 6. レスポンシブ（端末・向き対応）

横持ちを推奨、縦持ちはサポート（使えればよい・見やすさは犠牲を許容）。手牌は14牌横並びで横幅を要するため横向きが本命。回転を促すヒントは出さない（横持ちは使い慣れていれば自明とみなす）。

- 出し分けの軸（PC を巻き込まないため）：横持ちで縦が低い＝`max-height`（PC モニタも幅>高さで landscape 判定になるため `orientation` だと PC を巻き込む。縦が低いのは横持ちスマホ固有なので height で線を引く）。縦持ち＝`orientation: portrait`（PC・横持ちは landscape 判定で該当しない）。いずれも「低い／狭い画面用のルールを追加」する形にし、PC・横持ち base の CSS は据え置く（既存の見た目を崩さないことを最優先）。
- メイン（クイズ）：手牌が4択に重ならないことを最優先（盤面は潰さずスクロールに化けるフォールバック）。横持ちはキャラの取り分を減らして盤面に高さを回す。縦持ちは手牌を2段に折り返し、場情報バーは中央寄せをやめ左寄せ、立ち絵は非表示にして会話枠を全幅にする。
- 会話枠（ノベル風）：大きさは状況（解説バッジの有無など）で変えず一定に保つ（PC と同じ）。あふれる本文は枠内スクロール、操作ボタンは枠基準で固定。
- スタート／設定：入りきらないときはスクロールで全項目に届かせる（1画面に詰め込まない）。

実装（CSS の2層分離・デザイントークン）の置き場は [architecture §5](./architecture.md)。
