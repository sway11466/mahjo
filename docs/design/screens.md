# 画面仕様（screens）

アプリの画面・シーン構成と、各画面のレイアウト・表示要素・操作。機能仕様が主題。採点・ヒント・キャラの中身は各 doc（[scoring-rules](../spec/scoring-rules.md)／[hints](../spec/hints.md)／[character-guide](../characters/character-guide.md)）、型は [data-model](./data-model.md) を参照。純ビジュアル指針（配色・牌の描画・ハイライトの見せ方・アニメ）は [uxui.md](./uxui.md)。

「画面＝大きな場所、シーン＝画面内の状態」。盤面は出しっぱなしで、下の操作パネルがシーンで切り替わる、という作り。

## 1. 画面一覧と遷移

- スタート画面：キャラの立ち絵＋メニュー（練習開始／キャラクター選択／ルール設定／アプリ設定）。練習開始でモード（役／点数）を選ぶ。
- メイン画面：出題と学習。出題シーン・解説シーンを持つ。役一覧・点数表をオーバーレイで参照（feature-3/4）。
- キャラクター選択画面：キャラ選択＋成績（ハートレーティング）。
- ルール設定（`RuleSettings`）／アプリ設定（`AppSettings`）。

遷移：スタート ⇄ メイン（練習開始→モードを選んで入る／戻る）、スタート ⇄ キャラクター選択／ルール設定／アプリ設定、メイン ⇄ 各画面。設定の分類は §5（`RuleSettings`＝採点に効く／`AppSettings`＝UX）。

上記はすべて React アプリ（`app.html`）の中の画面。これとは別に、初見向けの**公開静的サイト**（LP・キャラ紹介＝素 HTML）がアプリの外側にある → §6。アプリへの入口は LP からのリンク（`/` → `/app.html`）。

## 2. スタート画面

学習の起点。選択中キャラの立ち絵で「いま誰と学ぶか」を見せる（背骨＝キャラ駆動）。

- 表示：選択中キャラの立ち絵（全身 `<id>-full-stand-a.webp`。背後に `themeColor` のグロー）と、右にタイトル（Mahjo）＋メニュー（練習開始／キャラクター選択／ルール設定／アプリ設定）。
- 練習開始：その場でメニューを役モード／点数モードに差し替える（インライン展開・別画面に飛ばさない）。選んで → メイン画面へ。役モード／点数モードの入口。
- レイアウト：広い画面は立ち絵を左・タイトル＋メニューを右、狭い画面は立ち絵→タイトル＋メニューの縦積み。メニューのボタンは同色（特定項目を強調しない）。

## 3. メイン画面

役モード・点数モードで共通の器（盤面・生成エンジン・成立役の説明文を共有）。モードで中身が変わる。遊び方は「解説（見て学ぶ）」と「クイズ（4択で試す）」。

画面の表示内容（盤面・キャラ・ヒント・4択・解説）は、提示層 `session` が1ターンごとに組み立てる view-state を ui が描画する（[architecture.md](./architecture.md) §2・[session.md](../spec/session.md)）。本章は「何が見えるか」を扱い、進行・判定・view-state の組み立ては session.md が正。

用語の区別：本章の「解説シーン」はクイズで回答した後に役/採点を見せる場面（クイズ session の一部）。遊び方の「解説（見て学ぶ）」は4択なしの単独モードで、これは別途設計（単発・役指定の見込み。session.md のスコープ外）。

盤面（両シーン共通の表示土台）：

- 手牌（和了形14枚／副露は副露面子を分けて表示）
- 上がり牌（どの牌で和了したか明示。ロン／ツモも明示）
- ドラ表示牌（裏ドラ表示牌はリーチ時のみ。赤ドラは設定オン時に区別表示）
- 場の状況：現在の局（東1局〜南4局。場風は局から導出）／自風（親・子）／ツモ・ロン／リーチ・一発・ダブルリーチ・海底河底 等のフラグ

### 出題シーン

- ヒントボタン：段階的ヒントを1段ずつ開く（答えは言わない・牌も指し示さない。設計は [hints.md](../spec/hints.md)）。現在のキャラの口調で表示する。ピンポイントのハイライトは回答後の解説シーン側。
- 遊び方「解説」：役モードは「役を表示」、点数モードは「採点説明」ボタン → 解説シーンへ。
- 遊び方「クイズ」：4択を出題（役モード＝成立役／翻、点数モード＝翻／符／点数）。回答 → キャラのリアクション → 解説シーンへ。

### 解説シーン

キャラが内訳を **1つずつ案内する**ウォークスルー。一覧をまとめて出すのではなく、成立役（→ドラ→〔点数モードは符〕）を順に、キャラのセリフ＋該当要素のハイライトで見せる。「答えに辿り着く過程の支援」（[product-concept](../product-concept.md)）をキャラ駆動で体現する。

- 入口（プレッシャーをかけない）：回答後、正誤（4択の中の ○/✗）とキャラのリアクションを見せた段で「解説を見る／次の設問へ」を選べる。解説は毎回提示するが強制はしない。
- 進行：「次へ」で1ステップずつ。複合役なら次の役へ、ドラがあればドラへ、全ステップを見終えたら「次へ」で次の設問へ。各ステップでキャラが説明セリフを話し、対応する盤面要素が光る。
- ハイライト連携：各ステップ（＝成立役／ドラ／符）が持つ `highlightTargets`（[data-model](./data-model.md) §9・§10）で、対応する画面要素（特定の牌・上がり牌マーカー・ドラ表示牌・場風/自風表示・門前ロン表示・千点棒 等）を光らせる。光らせ方はやわらかく（[uxui.md](./uxui.md) §3）。将来、項目クリックでの個別ハイライト併用も可。
- 説明セリフはキャラ別（`character-<id>-script.md` §3 解説セリフ）。回答後なので役名を出してよい。中身（何を説明するか）はエンジンの役判定／採点結果が正で、文言だけキャラ依存。
- 役モード：成立役（＋ドラ）を順に。点数モード：翻・符・点数も同様に1つずつ（符の発生源＝副底20・待ち2・暗刻4・雀頭2・ツモ2 等、最終点数＝例「子ロン 5200点」）。
- クイズ回答後（誤答）：選んだ誤答の `MistakeKind`（[data-model](./data-model.md) §12）に応じて**キャラがそっと諭す**（中立テキストの併記ではなく声で。文章は全てキャラの声＝一貫性／寄り添う）。セリフは「短い寄り添い（場面 wrong プール）＋ 具体の諭し（MistakeKind 別）」を結合（文言は `character-<id>-script.md` §1 と §4・基準は [hint-base.md](../spec/hint-base.md)「誤答の諭し素」、答えは言わない）。その後「解説を見る」で正解の内訳ウォークスルー＝「ミスの所在 → 正解の理屈」を順に示す。

### クイズ共通仕様（出題シーン⇄解説シーンにまたがる）

- セッション構成：1セッション＝8問（東南戦：東1局〜南4局）。盤面に現在の局を表示し、進捗（n/8）を兼ねる（場風チップは出さない）。場風は局から導出（東場=東／南場=南）。各問は独立に生成し、自風・親子は問題ごと。局番号は「場風＋進捗」の表示であって、親のローテーションは持たない（realに局で親を回す方式は将来検討）。
- 選択肢（4択）の配置：手牌と会話パネルの間に独立した4択バーを置く。会話パネルはヒント／リアクション専用に保ち、誤タップを避ける（4択を会話窓に兼ねさせない）。広い画面は横1列、狭い画面は2×2に折り返す。
- 選択肢の値の表示：数値で答える。役モードは複合役がありうるため役名でなく翻で答える＝「X翻」。点数モードは総合で「X翻Y符ZZZZ点」（符・点数を個別に問う場合は「Y符」「ZZZZ点」）。役そのものの確認は解説（フラッシュカード）／ヒント側に置く。
- 出題内容：役モード＝成立役／翻、点数モード＝翻／符／点数。何を問うかはモード／設定で選ぶ。
- 選択肢：正解＋誤答3の4択。誤答は「学習者が実際にやりがちなミス」（親子取り違え・ドラ見落とし・ツモ/ロン取り違え・符/翻の数え違い）をミス変換で生成し、各誤答に理由ラベルを持たせる。生成は engine 寄りの純ロジック（`mistakes`、テスト対象）。型は [data-model](./data-model.md) §12。
- リアクション：正誤に応じてキャラが反応（表情＝[data-model](./data-model.md) §13 Expression）。短い声かけで気分よく。
- 連続正解などのゲーム要素は任意（parking lot で検討）。

## 4. キャラクター選択画面

選択中キャラの立ち絵を大きく見せ、キャラ選択と成績を兼ねる。

- キャラ選択：使うサポートキャラを選ぶ（`AppSettings.selectedCharacterId`）。一覧（サムネ）から選んで切替（選択中はテーマ色のリングで示し、立ち絵・テーマ色も即追従）。選択は即保存・復元（既定はまお）。一覧はレジストリ駆動なのでキャラを足せば自動で並ぶ。
- 成績：「好感度」としてハート5つのレーティングで見せる（`Progress.correctTotal`〔キャラ別〕由来）。ハート1個＝10問正解相当、満タン＝累計50問。伸び中の1個は端数で部分点灯（例 32問＝3個点灯＋4個目が 2/10）。50問で上限（お祝い止まり・以降は満タン維持）。累計正解数の生の数字は出さない（ハートだけで好感度を示す）。正答率・ストリーク・未達警告も出さない（[product-concept.md](../product-concept.md) §3 プレッシャーをかけない）。

## 5. 設定（ルール設定・アプリ設定）

2画面。すべて即時反映・localStorage 保存。分類は型の区別に対応（`RuleSettings`＝採点に効く／`AppSettings`＝採点に効かない UX）。

### ルール設定（`RuleSettings`）

- 喰いタン・後付け・赤ドラ枚数・切り上げ満貫・数え役満・ダブル役満・レア役・場の固定/ランダム・出題する役の範囲。選択肢・既定は [scoring-rules.md](../spec/scoring-rules.md) §5。

### アプリ設定（`AppSettings`）

- 効果音（se）・音楽（bgm）・プレイヤーの呼び方（playerName）・牌のランダム並び（randomTileOrder）。キャラ選択（selectedCharacterId）はキャラクター選択画面（§4）が持つ。

## 6. 公開サイト（LP・キャラクター紹介）— 静的

アプリ（§2〜§5＝React・`app.html`）の外側に置く、初見向けの**素 HTML** ページ群。狙いは SEO と「最初の顔」（コンセプト・キャラ・使い方を伝える）。**本節がサイト構成・URL・内部リンク・各ページの中身・画像方針・SEO の正**。配置・組み立ての機構（`app.html` 分離・`public/` 素通し・PWA `start_url`）は [architecture.md](./architecture.md) §4、デプロイ・base は [development.md](../dev/development.md)「デプロイ／ブランチ戦略」。feature-5（[backlog](../backlog.md)）。

世界観（[world.md](../characters/world.md)）のキャストは継続的に増える（まお・りん＋姉弟子・大師匠・両師匠……確実に5人以上）。この規模を前提に、キャラ紹介は **ハブ&スポーク**（一覧ページ＝ハブが各紹介＝スポークへ権威を配る）で組み、一覧ページをサイトの背骨に据える。

### ページと URL

`public/` 配下はファイルパスがそのまま URL（base 付与時は各 URL に base〔例 `/mahjo/`〕が前置）。

| URL | 実体 | 内容 |
|---|---|---|
| `/` | `public/index.html` | LP（素 HTML） |
| `/characters/` | `public/characters/index.html` | キャラ一覧（ハブ。世界観の入口＋各紹介への分配点） |
| `/characters/<id>/` | `public/characters/<id>/index.html` | キャラ紹介（キャラごとに独立 URL＝SEO） |
| `/privacy/` | `public/privacy/index.html` | プライバシーポリシー（兼免責。アクセス解析・クッキー・端末内保存・免責） |
| `/app.html` | React アプリ | アプリ本体（§2〜§5） |

### 内部リンク（ハブ&スポーク・向きが命）

SEO で効く一覧ページは「**強いページから貼られ、子へ権威を配る分配点**」。リンクの向きを次の通りにする（子→親だけにしない）：

- **LP → ハブ**：LP のグローバルナビ「キャラクター」と、キャラ節末尾の「全員を見る →」を **`/characters/`** に向ける（LP 内アンカー `#characters` 止まりにしない）。一番被リンクの集まる LP からハブへ権威を流す。
- **ハブ → 各キャラ**：ハブの全キャラカード／相関図ノードから各 `/characters/<id>/` へ。
- **各キャラ ↔ 各キャラ（相互リンク）**：相関（[world.md](../characters/world.md) §5）に基づく関連キャラ（同門・師弟・ライバル）へ横リンク。単純な前後送りより、相関ベースの関連リンクを優先（設定を活かし文脈も強い）。
- **各キャラ → ハブ／LP**：パンくず（下記「グローバルナビ／パンくず」）で `Home › キャラクター › <名>` を辿れる。

### ページの役割分担（薄い重複ページを作らない）

同じ「キャラ」を扱う3面が内容を食い合わないよう、固有の価値を分ける：

| ページ | 固有の価値（これが無いと薄い重複になる） |
|---|---|
| LP `#characters` | **ダイジェスト**。世界観1〜2文＋代表キャラ2〜3人＋「全員を見る →`/characters/`」。全員は載せない（ハブへ送る）。 |
| ハブ `/characters/` | **キャストの索引＋世界観の入口**。①世界観リード文（[world.md](../characters/world.md) §1〜4 由来）②相関図（[world-relationships.svg](../characters/world-relationships.svg) を掲載）③館・系譜でのグルーピング（館名は TBD のため館の説明を見出しに＝由緒ある／繁盛する 等。大師匠系＝world.md §4・§5）④全キャラカード。LP にも詳細にも無い固有コンテンツを持たせ、「Mahjo のキャラ世界」の検索意図を受ける。 |
| 各 `/characters/<id>/` | そのキャラの深い**読み物**（世界観・性格・使い魔・ビジュアル）。内容の正は `docs/characters/<id>/character-<id>.md`（紹介文はそこからの抜粋＝設定を二重管理しない）。「このキャラと学ぶ」で `/app.html` へ（キャラ指定の深いリンクはルーティング未整備＝今後・[backlog](../backlog.md) parking lot）。 |

- **アプリ内のキャラクター選択画面（§4）とは別物**：§4 は機能（選択＋成績ハート・React）、本節は公開・SEO 用の紹介（静的）。

### グローバルナビ／パンくず（全ページ統一）

- **ナビ統一**：全静的ページのヘッダーを同一構成にする＝ブランド／特徴／学び方／**キャラクター（→`/characters/`）**／アプリを開く。「キャラクター」は常に同じ URL（ハブ）を指す（同じ語が同じ先＝クロールも UX も一貫）。
- **パンくず**：各下層ページにパンくずを置き、階層を明示する（クロール経路＋下記 SEO の `BreadcrumbList`）。階層はページ依存（各キャラ＝`Home › キャラクター › <名>`、`/privacy/`＝`Home › プライバシーポリシー`）。
- **フッタ**：LP フッタに `/privacy/`（プライバシーポリシー）へのリンクを置く（法務系リンクの定位置）。プライバシーポリシーは国内向けの表記対応で、Google アナリティクス／クッキーの利用・端末内保存（localStorage）・第三者（Google）・免責を記載。クッキー同意バナーや独立した利用規約は当面持たない（国内中心の前提）。
- **定型ブロックは手で揃える**：各ページは手書きで個別最適する（自動生成しない＝下記「生成しない」）。ヘッダー・フッター・パンくず・カードの**定型マークアップだけ**は1つのコピー元に揃え、ズレを防ぐ（スタイルは `site.css` で一元化済み）。

### 画像方針（A）

1 master → 用途ごとに書き出し（[character-guide](../characters/character-guide.md) §3）に乗せる。アプリ内画像とサイト用画像は同じ master からの別 export であり、二重管理（ドリフト）ではない。

- アプリ内＝`src/assets/**`（バンドラ経由・ハッシュ付き・最適化）。
- サイト用＝`public/img/**`（素 HTML から固定パスで参照）。立ち絵は master から流用、OGP/SNS 共有画像（1200×630 等）はサイト専用に書き出す。
- 素 HTML は `src/assets` のハッシュ付き出力を安定参照できないため、サイトが使う画像は `public/` に置く（＝この方針の前提）。

### SEO

各素 HTML が持つメタ：`<title>`／`<meta description>`／OGP・Twitter Card（`og:title`/`og:description`/`og:image`/`og:url`/`twitter:card`）／`<link rel="canonical">`。`robots.txt`・`sitemap.xml` は `public/` 直下（固定パス）。`og:image`・`canonical` の絶対 URL は `mahjo.academy`（2026-06-20 移行済み）。

JSON-LD（構造化データ）をページ種別ごとに持つ：

| ページ | JSON-LD |
|---|---|
| LP `/` | `WebApplication`（＋任意で `WebSite`） |
| ハブ `/characters/` | `CollectionPage` ＋ `ItemList`（各キャラを `itemListElement` に列挙） |
| 各キャラ `/characters/<id>/` | `ProfilePage` ＋ `Person`（`name`/`description`/`image`/`url`。相関は `colleague`／`knows` で関連キャラへ＝[world.md](../characters/world.md) §5 を構造化） |
| 全下層ページ | `BreadcrumbList`（パンくずと対応） |

- **sitemap**：キャラ追加ごとに `/characters/<id>/` の行を足す（手書き）。優先度の目安＝LP 1.0／ハブ 0.7／各キャラ 0.5。

### 生成しない（手書きで個別最適）

キャラ紹介ページは「最初の顔」かつ SEO の主戦場で、キャラごとに見せ方の最適が異なる（小悪魔／素直 等）。テンプレからの自動生成はしない ── 生成の型に合わせて紹介文・見出し・OGP・相関の語り口を書く制約がキャラ制作（[character-guide](../characters/character-guide.md)）に混入し、キャラ差を均してしまうため。**各ページは手間を惜しまず1枚ずつ手書きで個別最適**し、定型ブロック（ヘッダー・フッター・パンくず）だけ手で揃える（上記「グローバルナビ／パンくず」）。ズレが実害になったら本書冒頭の方針（[architecture](./architecture.md) 冒頭「意図の読み取りやすさ・シンプルさ」）に照らして再考する。

### スタイル

純ビジュアル指針（配色・トーン）は [uxui.md](./uxui.md) に準じ、素 HTML 共有 CSS（`public/site.css`）で当てる（`src/ui/index.css` のトークンはバンドラ越しに import できないため、サイト用に必要な値を持つ）。
