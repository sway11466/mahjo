# 出題生成（役シード方式）

ランダムな和了形を出題するための生成設計。役判定・符・点数の正は [scoring-rules.md](./scoring-rules.md)、型は [data-model.md](../design/data-model.md)、生成器のテストは [testing.md](../dev/testing.md) §7。生成は engine の純ロジック（`src/engine/generate`、`rng` 注入で決定的）。

## 1. 役シード方式（手順）

1. 出題対象の役プール（§3 で決まる）から役を1つランダム抽選＝シード役。
2. シード役を必ず含む合法和了形を構築。役ごとに「構築器」を用意（例：清一色→1色の数牌で4面子1雀頭、対々和→刻子4＋雀頭、三色同順→同じ並びの順子を3色）。
3. 制約を守る：門前役（ピンフ・一盃口・リーチ・門前ツモ 等）は門前で構築。各牌は4枚まで。么九/中張など役の要件を満たす。
4. 残りの自由部分はランダムに埋める。
5. 和了牌・ツモ/ロン・場風/自風・ドラ表示牌・リーチ有無などを付与。場風はパラメータで渡されたらそれを使用し、なければランダム（`RuleSettings.round` に従う。§2）。他は設定に従い必要に応じてランダム。
6. 完成手を採点エンジンに通し、実際に成立する全役・符・点数を算出して表示する。

シード役は「生成のきっかけ」にすぎない。表示する採点結果はエンジンが実際に検出した内容を正とする（シード以外の役が複合してよいし、シードが上位役に吸収されてもよい。例：一盃口→二盃口）。原則の詳細は [architecture.md](../design/architecture.md) §3。生成は通常役シードのみ、役満シードは parking lot（未対応）。

## 2. 場風・自風（セッション連携）

クイズは1セッション＝8問（東南戦：東1局〜南4局。[screens.md](../design/screens.md) §3）。局が場風を決める（東1〜4局＝東場／南1〜4局＝南場）ので、生成は場風を入力で受け取れるようにする。

- 場風がパラメータで渡されたら、それを使う（その局の場風に整合する手を作る。例：`yakuhai-round`〔場風牌の刻子〕は渡された場風牌で構築する）。
- 渡されなければランダム（`RuleSettings.round`：`east-fixed`＝東固定／`random`＝東・南からランダム）。これがセッション未接続の単発生成の挙動。
- セッション接続後は、現在の局の場風を生成に渡し、その局に整合した手を出す。

自風・親子は当面、問題ごとにランダム（局による親のローテーションは持たない。[screens.md](../design/screens.md) §3）。将来 real に局で親を回す場合は、自風も同様にパラメータ化する。

### 和了状況の付与（ドラ・副露・リーチ）

1手だけでは決まらない要素（副露・リーチ等）も、現実の和了状況に近づけるため確率で付与する。フラグの**依存関係（順序）が重要**：

1. **副露（ポン・チー・明カン）**：その役で可能なら一定確率で1面子を明にする（喰い下がり〔`hanOpen`〕を出題に出すため）。門前限定役（平和・一盃口・二盃口・七対子）や、副露で形が崩れる役（三暗刻）は副露しない。喰いタンなしの断幺九も副露不可（消えるため）。明カンは符の話なので当面オフ（暗カンも見送り）。
2. **リーチ**：**門前のときだけ**（副露したら不可＝門前必須）。一定確率で付与。リーチ時のみ**裏ドラ表示牌**を1枚出す。
3. **一発**：リーチのとき更に低確率で付与。
4. 門前ツモは「門前×ツモ」から採点側が導出するので、副露すれば自動的に外れる（フラグ不要）。

ドラ表示牌は常に1枚出す（実際の卓と同様）。表示牌は手牌と物理 id が衝突しない牌を使う。海底・河底・嶺上・槍槓・ダブルリーチ・天和・地和は当面オフ（高頻度だと不自然・レア状況）。

確率の既定（仮・`engine/generate.ts` の `P_OPEN` / `P_RIICHI` / `P_IPPATSU`。プレイ感で再評価）：

| 事象 | 確率 | 条件 |
|---|---|---|
| 副露 | 40% | 副露可能なシードのみ |
| リーチ | 30% | 門前手のみ（副露と排他） |
| 一発 | 5% | リーチ時のみ |

## 3. 難易度と漸進的な出題範囲

役には「理解（見抜き）の難易度」がある。採点の難しさ・翻の高さとは別軸（例：清一色は高得点だが一色なので見抜きは易しい／平和は複合条件を満たす形を見抜く必要があり難しい）。学習が進むほど、出題に使う役を易→難へ広げる。

- 出題プール ＝ `RuleSettings.enabledYaku`（設定でオン）∩ アンロック済みの難易度帯。
- 現在のキャラの成績（`Progress`、キャラ別。[data-model.md](../design/data-model.md) §16）の節目で次の難易度帯が解放され、出題役が増える。キャラを切り替えると、そのキャラの成績・難易度帯で続く。
- 増える方向のみ・減点や逆戻りなし（プレッシャーをかけない。[product-concept.md](../product-concept.md)）。キャラの表情/衣装アンロックと同じ「お祝いで解放」の思想（[character-guide.md](../characters/character-guide.md)）と整合。

難易度の割り当て（仮・要精査）：

| 帯 | 例 |
|---|---|
| 易 | 七対子・断幺九・役牌・対々和 |
| 中 | 三色同順・一気通貫・混一色・清一色・一盃口 |
| 難 | 平和・チャンタ・三暗刻・二盃口 |

役→難易度の対応は本書（上表）で持つ（data-model には型化しない）。実装は `engine/generate.ts` の構築器登録（各シード役に `band`）が上表に対応し、未対応の役は出題プールに入らない（構築器が増えるとプールも増える）。

決定（仮・要精査）：

- 帯は3段（易/中/難）。アンロックは**モード別**＝ `Progress.correctByMode[mode]` で駆動（役モードの進捗が役モードの難易度を、点数モードの進捗が点数モードの難易度を上げる）。
- アンロック閾値（仮）：**中＝10問・難＝30問**（そのモードの累計正答数）。`engine/generate.ts` の `BAND_THRESHOLD` が正で、調整しやすいよう定数化。値はプレイ感を見て再評価する。
- 出題プール ＝ `enabledYaku`（オン） ∩ 解放済み帯 ∩ 構築器のある役。増える方向のみ・逆戻りなし。
- 「進捗を使わず全役出題」専用の設定は当面持たない（全役を試したい場合は `enabledYaku` と帯解放で足りる。必要になれば再検討）。

### 複合役と難易度（生成後ガード）

シード役は下限の保証にすぎず、残りの面子をランダムに埋めるため**複合役が創発する**（generation.md §1 と整合：シード以外の役が複合してよい）。このため易シード（例：断幺九）でも、ピンフ・三暗刻などの上位帯の役、まれに役満が混ざり、**実現難易度が解放帯を超える**ことがある（実測：断幺九シードの約1/3で易を超える形役が複合）。

方針（C+A）：

- **生成後ガードを1回だけ発動**。生成した手の「実現難易度」（検出された形役の band の最大、役満は最上位超え）が解放帯を超えたら、同じシードで**1回だけ振り直す**。`engine/generate.ts` の `generate()` が担当（`realizedRank`）。
- **なお超えても許容**する（厳密保証はしない）。複合役は学習上むしろ歓迎で、振り直しを繰り返すと生成が重く・分布が歪むため。構造的に上位役を伴いやすいシード（例：対々和はほぼ三暗刻を伴う）は許容側に倒れる。
- 判定対象は構築器を持つ役（band 既知）＋役満。band 未定義の非シード役（三色同刻・混老頭 等）はガード対象外（まれなので許容）。
- 低レベルの `generateForSeed` は純粋な構築のみでガードを持たない（読みやすさのため、難易度方針は `generate` に集約）。
