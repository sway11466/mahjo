# ヒント設計

「答えを言わずに、自力で気づかせる」段階的ヒントの作り方・管理方針。**素（中立）と各キャラのセリフ（全量）を分けて管理**する。型は [data-model.md](../design/data-model.md) §11、素は [hint-base.md](./hint-base.md)、キャラ別セリフは `character-<id>-script.md`、テストは [testing.md](../dev/testing.md)（ヒント・クイズのテスト）。

## 1. 位置づけ

- 段階的に「ぼんやり→具体」へ小出しし、自力の気づきを促す。確定的な答えは「役を表示」「採点説明」側に置く。Mahjo の堀（[product-concept](../product-concept.md)）。
- 解説・クイズ両方／役・点数モード両方で使う。
- アプリは文章を**生成しない**。authored なセリフを**選択・組み立て**するだけ。
- 担い手は使い魔（familiar）。ヒントボタンを使い魔の見た目にし（「ヒント」の文字は残す）、セリフは「使い魔がそっと示唆 → キャラ本体がひらめいて言葉にする」語り口で書く（A方向＝[decisions.md](../decisions.md) 2026-06-10）。声の主はキャラ本体のまま（使い魔は声を持たない＝authoring は1人分）。使い魔の所作は地の文に書かず視覚（ボタン／演出）で表す。使い魔ビジュアルは素材待ちで、未配置の間はテキストの「ヒント」ボタンにフォールバック（[character-guide.md](../characters/character-guide.md) §2「使い魔」）。

## 2. 二段構え：素（中立）とキャラ別セリフ（全量）

| 層 | ファイル | 中身 |
|---|---|---|
| ヒント素（正本） | [hint-base.md](./hint-base.md) | 着目ポイントごとの段階＋中立の意図。**全役・全符を網羅**。キー＝エンジン語彙（`yaku:*`/`fu:*`/`dora`）。何を気づかせるかの土台＝正確性・カバレッジ担保 |
| キャラ別セリフ（正本） | `character-<id>-script.md` | 各キャラが hint-base の**全キーを自分の言葉で網羅**。個性（好きな役・口調）はここ |

突き合わせ：script のキー集合 == hint-base のキー集合（過不足なし）を機械チェック。**全量保証なのでフォールバック不要**。これにより「素＝網羅・正確」「キャラ＝個性」を両立する（語尾を包むだけでは出せない味＝チートイ好き／ツンデレの「これくらい余裕」等も出せる）。

## 3. 着目ポイント × 段階

ヒントは着目ポイント（学習対象の最小単位）に紐づく。`level 0`（ぼんやり・どこを見るか）→ `level N`（具体・成立の一歩手前）。**答え（役名・確定値）は含めず、牌も指し示さない**（ピンポイントのハイライトは回答後の解説側＝[screens.md](../design/screens.md) §3）。採点 `ScoreItem` と同じ着目ポイント軸で揃う。具体例は [hint-base.md](./hint-base.md)。

加えて、特定の役・符によらない**汎用ヒント（`generic`）**を持つ。手牌全体・牌の色・あがり牌周辺などを促す緩い気づきのプール。**盤面表示後しばらくで自動表示（idle／時間トリガー）**したり、段階ヒントの最初のぼんやり段として使う。プレッシャーをかけず寄り添うための“そっと促す”層。

## 4. 実行時（二層：HintProvider → HintRenderer）

文言の生成はしない。第1層が**どこを気づかせるか**の骨組みを選び、第2層が**現在キャラの手書きセリフ**を差し込む。キャラ依存は第2層だけ（型は [data-model.md](../design/data-model.md) §11）。

第1層 `HintProvider`（キャラ非依存・テスト対象）── 入力：`hand`/`table`/`winContext`/`ScoreResult`/`mode`。

1. エンジン検出の役・符（`ScoreResult.items`）から該当する着目ポイントを**選択**し、順序・`level` を決める。
2. 文言を持たない骨組み列 `HintStepPlan[]`（`key`／`level`）を「ぼんやり→具体」順で返す（牌は指し示さない）。

第2層 `HintRenderer`（キャラ依存はここだけ）── 入力：`HintStepPlan[]` ＋ 現在キャラの `HintScript`。

4. 各 `plan.key`＋`level` で現在キャラ script から手書きセリフを引き、`HintStep`（表示用）を完成。
5. UI がボタンで1段ずつ開く。

役モード＝認識（見抜く）、点数モード＝計数（数える順のウォークスルー）。

## 5. テスト（[testing.md](../dev/testing.md)）

- script のキー集合 == hint-base のキー集合（書き漏れ・余剰の検出）。
- 段が `level` 昇順。答え（役名・確定値）を `text` に含めない。
- 場面セリフは場面セットを網羅。

## 6. 順序ポリシー（決定）と未決事項

**選択・順序（決定）**：学びの体感を優先した手書きの優先順（スマートな式は使わない）。`engine/hints/provider.ts` の `SPOT_RANK` が正。

- **役モード**：状況役（リーチ/一発/門前ツモ…）→ ひと目で分かる役（七対子・清一色・混一色・対々和・字一色・国士…）→ 易（断幺九・役牌）→ 中（三色・一通・一盃口）→ 難（平和・チャンタ・三暗刻…）→ **ドラ（最後）**。**1着目ポイント＝1段**（level 0）。冒頭の汎用ぼんやり段は置かない（盤面提示時に同等のセリフが出る前提）。
- **点数モード（暫定）**：役（上と同順）→ 符（副底→門前ロン/ツモ→待ち→雀頭→面子 の数え順。`computeFu` の積み順をそのまま）→ ドラ。数え順の詳細は今後精査。

未決事項（案つき）：

- **着目ポイントのキー型**：符発生源の識別子（`FuSource` 等）を [data-model.md](../design/data-model.md) に型として置くか。役は `YakuId` を流用。現状の符キーは `fu:base`/`fu:wait`/`fu:meld` 等を provider が item id から導出。
- **idle トリガー**：汎用ヒントの自動表示（盤面表示後の経過時間）をどう持つか（UI タイマー／`ReactionTrigger` に `idle` を足す 等）。発火間隔も要調整。
- **場面語彙の統一**：セリフの場面と表情トリガー（`ReactionTrigger`）を共通語彙に揃えるか。
- **段数・粒度の設定可否**（任意）。
