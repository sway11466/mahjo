# ADR-0003: BGM はコード生成方式（Web Audio 合成・2層）で実現する

- ステータス: Accepted
- 日付: 2026-07-11
- 決定者: （プロジェクトオーナー）
- 関連: [sound.md](../design/sound.md)、[architecture.md](../design/architecture.md)（§1 技術スタック・§2 レイヤ分離）、[character-guide.md](../characters/character-guide.md)、[ADR-0001](./ADR-0001-no-state-management-library.md)

---

## Context（背景）

Mahjo はバックエンドを持たない完全静的な PWA で、初回読込後はオフライン動作する（[architecture.md](../design/architecture.md) §1）。学習中ずっと鳴らしうる BGM（既定オフ）を、どう用意するかの岐路に立った。取り得る方式は概ね3つ：

1. 音源ファイルを同梱（mp3/ogg 等をループ再生）。
2. 音楽ライブラリ（Tone.js 等）で合成・シーケンス。
3. 素の Web Audio API で自前にコード生成（合成）。

判断材料になる制約：

- 配布サイズ ── PWA は全アセットを precache するため、ループ用の音源ファイルはオフライン容量にそのまま乗る。キャラが増えるほど曲も増える。
- 権利 ── public リポジトリ＋静的配信があり得るため、音楽の再配布ライセンスを持ち込みたくない（SE で「権利表記をしない」方針を採ったのと同じ理由＝[sound.md](../design/sound.md)「ライセンス方針」）。
- キャラ追加の軽量運用 ── 「キャラ追加＝データ＋画像だけ、ロジック不変」（[character-guide.md](../characters/character-guide.md)）を BGM でも守りたい。曲がバイナリ音源だと「データ」が重くなる。
- 依存 ── ミニマル方針（[CLAUDE.md]）。ランタイム依存は積極的な理由がなければ増やさない（[ADR-0001] と同じ姿勢）。

---

## Decision（決定）

BGM は**素の Web Audio API によるコード生成**で鳴らす。音源ファイルも Tone.js 等の音楽ライブラリも使わない。曲は**2層構成**で持つ：

- **主旋律（しゅせんりつ）** … キャラごとの決まった旋律。**度数記法**（度数 1〜5・オクターブ `'`/`,`・長さ `:拍`・休符 `0`・空白区切り）で1本の文字列として書く。楽器・主音・モード（調）・テンポ・明るさ・余韻をあわせて持つ。
- **即興音（そっきょうおん）** … 主旋律の合間を飾る、毎回ちがう生成音。**セグメント配列**（`{ start, length, instrument, density, octave, offset, volume, brightness, sustain }`）で「どの区間を・どの楽器で・どれくらいの密度で」だけを指定し、実際の音は再生時に生成する。

記法・楽器・モードの語彙と各フィールドの意味・取り得る値の正は [sound.md](../design/sound.md)「BGM の実現方式」。

層の分担（[architecture.md](../design/architecture.md) §2 の依存方向に従う。SE と同じ音 IO）：

- **データ（何を鳴らすか）は characters 層** … キャラ固有の主旋律・即興音は `Character` のデータとして持つ（キャラ別データの正は `character-<id>-sound.md`）。characters は DOM も Web Audio も持たない（文言・参照データのみ）ので、ここは「楽譜」に相当するデータだけ。
- **合成・スケジューリング（どう鳴らすか）は ui 層** … 度数記法のパース・Web Audio による音色合成・小節スケジューリング・再生/停止は ui が担う（牌 SVG・SE と同じく描画/再生の IO はここに集約）。engine（麻雀計算）には一切入れない。

オーサリング（作曲・試聴）は開発ツール [tools/melody-authoring](../../tools/melody-authoring/) で行う（アプリ本体・公開サイトには同梱しない独立物）。記法パーサと合成ロジックは本番 BGM 機能（`src/`）と重複するが、現時点では無理に共通化せず、本番実装時に共有ロジックを抽出する。

## Consequences（結果・影響）

### 正の影響
- **配布サイズが実質ゼロ** … 曲は文字列（楽譜データ）だけ。音源バイナリを precache しないので、キャラが増えても PWA のオフライン容量が膨らまない。
- **権利の問いが消える** … 自前合成なので、音楽ライセンス（表記・再配布）の判断を持ち込まなくてよい。public リポでも気にしなくてよい。
- **キャラ追加＝データだけ** … 曲がデータ（主旋律の文字列＋即興音の配列）なので、「キャラ追加はデータ＋画像、ロジック不変」の約束を守れる。
- **ランタイム依存ゼロ** … 素の Web Audio API のみ。ビルドにも配布にも音楽ライブラリを足さない。

### 負の影響・トレードオフ
- **音の表現に限界** … 合成音は生録音・本格 DTM の質感には届かない。環境音的ループ（集中を妨げない）という BGM の役割（[sound.md](../design/sound.md)「方針」）に用途を絞ることで見合わせる。
- **ロジックの重複** … 記法パーサ・合成ロジックがオーサリングツールと本番機能で二重に存在する（本番実装時に共有ロジックへ抽出する前提で、当面は許容）。
- **オーサリングに手間** … 曲を「書く」道具（度数記法）と耳での試聴が必要。専用ツール（melody-authoring）で担保する。

---

## Alternatives Considered（検討した代替案）

### 音源ファイルを同梱（mp3/ogg ループ）
- 質は高いが、precache でオフライン容量に乗り・キャラごとに増え・再配布ライセンスの判断が要る。サイズ・権利・軽量運用のいずれにも逆行。→ 不採用。

### Tone.js 等の音楽ライブラリで合成
- コード生成という方向は同じだが、ランタイム依存が増える。素の Web Audio で二層構成は成立し、依存を足す積極的理由が弱い（[ADR-0001] と同じ判断）。→ 不採用。必要が生じたら別 ADR で再検討する。
