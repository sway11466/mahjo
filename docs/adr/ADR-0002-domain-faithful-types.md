# ADR-0002: 型は現実の麻雀ドメインを忠実にモデル化する（採点スコープ内で）

- ステータス: Accepted
- 日付: 2026-06-05
- 決定者: （プロジェクトオーナー）
- 関連: [data-model.md](../design/data-model.md)、[ADR-0001](./ADR-0001-no-state-management-library.md)

---

## Context（背景）

データモデルの型を、アプリの画面・実装の都合に合わせた便宜的な形で組むか、現実の4人打ち麻雀の構造に沿って組むかという方針の岐路に立った。

検討の中で、自然と「現実の麻雀の状況をそのまま型で表す」発想に寄っていることに気づいた。例：

- 牌を「現実の雀牌セット（136枚の物理牌）」として捉え、id を正準順の通し番号にする。
- 「場（テーブル全体で共有）」と「自家（自分の席・手）」は現実で分かれているので、型でも分ける。
- 「手牌（盤面に見える手）」と「点数計算の解釈（面子分解）」は別物なので、型でも分ける。

一方で、本アプリのスコープは「1回の和了を採点する」ことであり、対局進行（山・捨て牌・全員の手・ツモ切り等）のシミュレーションは非ゴール（[product-concept](../product-concept.md)）。

---

## Decision（決定）

データモデルの型は、アプリ都合の便宜的な形ではなく、現実の4人打ち麻雀のドメイン構造を忠実に写す。ただし範囲は「1回の和了の採点に関わる構造」に限定する。

具体的な現れ方:

- 牌＝現実の136枚の物理牌。id は正準順の通し番号（識別・並び順を兼ねる）。模様(suit/rank/honor)は id から導出。
- 「場（Round）」「手牌（Hand）」「自家の和了状況（WinSituation）」を別の型に分け、入力（WinContext）がそれらを内包する。
- 「手牌（盤面）」と「点数計算の解釈（Decomposition＝面子分解＋待ち）」を別の型に分ける。
- 導出できるもの（親/子＝自風が東か、並び順＝idやsuit/rankから）は保持せず導出する。

スコープの歯止め:

- 採点に関わる現実構造は忠実にモデル化するが、ゲーム全体のシミュレーション（山・捨て牌・全員の手・進行）には踏み込まない。
- 「採点に効くが進行依存」の要素は是々非々で判断する（例：本場・供託は点数表示に効くので将来取り込みうる／山・捨て牌は持たないので流し満貫は別扱い）。

---

## Consequences（結果・影響）

### 正の影響
- 現実の麻雀知識がそのままコードの構造に対応し、読みやすい。役判定・符計算を現実の概念単位（面子・場・自家）で書ける。
- 不正な状態を型で排除しやすい（判別共用体・必須フィールド化など）。
- ドメインが安定（麻雀のルールは変わらない）なので、型の寿命が長い。

### 負の影響・トレードオフ
- アプリの表示に最適化した形へは、ドメイン型から都度変換が要ることがある。
- 模様を id から導出するなど、利便性のための「派生」を持つ場面では、唯一の出所（正準テーブル＋ファクトリ）を守る規律が要る。
- 「採点スコープ内かどうか」の線引きを時々判断する必要がある。

---

## Alternatives Considered（検討した代替案）

### アプリ画面都合の型（UIに必要な形だけ持つ）
- 実装着手は早いが、現実とズレて役・符ロジックが歪み、保守しづらくなる。→ 不採用。

### 完全なゲームシミュレーションの型化（全員の手・山・進行まで）
- 非ゴール（対局進行はしない）に反し、過剰。→ 不採用。
