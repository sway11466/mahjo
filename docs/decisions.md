# 決定の一時置き場

書き場所がまだ決まっていない決定事項を、落とさないよう一時的に書き起こすインボックス。各決定は、収まる doc（product-concept・各設計doc・ADR 等）が定まったらそこへ移し、ここから消す。目標は空にすること。

恒久的な決定ログではない。重要な技術判断の恒久記録は `docs/adr/`。各エントリはここにある間は 決定／理由／影響 の数行で書き、新しいものを上に足す。

---

## 2026-06-10 ヒントは「使い魔の示唆 → キャラのひらめき」で出す（A方向）

- 決定：ヒントの担い手として使い魔（familiar）を初めて UI に出す。方式は「使い魔は声を持たない相棒（演出担当）」＝Fork A。具体的には (1) ヒントボタンを使い魔（まお＝猫）の見た目にする（「ヒント」の文字は残す）、(2) ヒントのセリフ（`HintScript`）を「使い魔がそっと示唆 → キャラ本体がひらめいて言葉にする」語り口に書き換える。会話窓の話者はキャラ本体のまま（声を増やさない＝authoring は1人分）。ヒント中の表情は既定どおり `insight`（ひらめき）で、この語り口と整合する。
- 理由：(1) 魔女＋使い魔の世界観に「使い魔がささやく」が噛み合い、ヒント＝「答えを言わず“そっと促す”」層と役割分離できる（[product-concept](./product-concept.md) §3）。(2) Fork B（使い魔が独自の声でヒントを語る）は型・アセット・script が使い魔ぶん増え、キャラ追加ごとに authoring が2倍になる。A は「キャラ追加＝データ＋画像、ロジック不変」（[characters/character-guide.md](./characters/character-guide.md)）の約束を守れて可逆。(3) 既存の `hinting:insight`（キャラのひらめき顔）と「キャラがひらめく」語り口が一致する。
- 影響：内容は各正本へ移行済み——語り口（旧・未決 (b)）は確定し、[spec/hints.md](./spec/hints.md) §1・[characters/character-guide.md](./characters/character-guide.md) §2「使い魔」＋各 `character-<id>-script.md` §2 に反映済み（まお／りんの script は「使い魔の示唆 → キャラのひらめき」で書き換え済み）。ヒントボタンの UI 枠も実装済み（使い魔画像を渡す口＋未配置時はテキスト「ヒント」フォールバック）。残る未決は (a) 使い魔の画像アセット未制作のみ（[backlog](./backlog.md) parking lot「使い魔アイコンの制作・配置」で追跡）。素材が入って未決が無くなったら本エントリは消す。

## 2026-06-06 Tailwind は当面入れず、素のCSS＋CSS変数で進める

- 決定：スタイルは素の CSS ＋ CSS カスタムプロパティ（デザイントークン）で実装する。Tailwind 等のユーティリティCSSフレームワークは現時点では導入しない。設定・クイズ・キャラ等の HTML 画面が増え、レイアウトユーティリティの反復が負担になった段で導入を再評価する。
- 理由：(1) 画面の主役が SVG 牌描画で、Tailwind の旨味（HTMLレイアウトの量産）が出にくい。man/pin/sou のような意味を持つ色は CSS 変数／セマンティッククラスが適する。(2) まだ画面数・トークンが少なく、ツール一式（依存・config・content スキャン）の導入はミニマル方針（CLAUDE.md）に対して重い。(3) Tailwind は後付けが容易で破壊的でないため、今コミットする必要がない。
- 影響：当面 `src/ui` は `index.css` ＋（必要ならコンポーネント単位の CSS）で構成。スタイル規約は [architecture.md §5](./design/architecture.md)。Tailwind 導入時はレイアウト系を TSX のユーティリティへ寄せ、装飾トークンは CSS 変数のまま残す方針（規約の2層分割は維持）。

## 2026-06-05 対象ユーザーは「作者を主対象・将来は歓迎」

- 決定：主対象は作者自身（麻雀の役・点数計算を体系的に学びたい人）。当面は自分の学習体験を最優先に設計判断する。一方で自分専用に閉じる設計はせず、同じ学習目的の人が使える余地は残す。
- 理由：今の目的は個人の学習。ただし将来的に利用者が増えるのは歓迎で、それを妨げる作り（特定環境への依存など）は避けたい。
- 影響：当面はバックエンドなし・localStorage 永続化のまま（マルチユーザー化は現時点の非ゴール → [product-concept](./product-concept.md) §4・[architecture](./design/architecture.md)）。「対象ユーザー」を独立セクションで持つのはやめ、本ログの判断として記録する。
